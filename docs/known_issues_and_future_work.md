# 已知问题与后续迭代方向

本文档记录当前项目在证书引擎兼容、部署环境和后续演进上的已知问题，作为后续迭代依据。

## 当前实现概述

当前项目已支持两种证书引擎：

- 首选：自安装或宝塔环境中的 `acme.sh`
- 兜底：项目内置的 Python ACME 引擎

## 已知问题

### 1. 内置 Python ACME 引擎是项目最小实现

当前内置引擎已经实现了 Cloudflare DNS TXT 验证和 ACME 证书签发，但它仍然是围绕本项目场景的最小实现。

这意味着：

- 功能范围聚焦在 Cloudflare Token 模式
- 当前未覆盖更多 DNS 提供商
- 当前不会为 `dns_ali` / `dns_dp` 提供内置引擎兜底
- 当前未实现更复杂的账户迁移和多目录管理能力

### 2. `acme.sh` 仍然是推荐首选

`acme.sh` 经过长期使用验证，在下列方面仍然更成熟：

- 边界场景处理
- 账号和证书目录管理
- DNS 提供商兼容性
- 社区验证与维护广度

因此项目当前策略是：

- 只要检测到可用的 `acme.sh`，就优先使用 `acme.sh`
- 只有在 `acme.sh` 不可用且 `ACME_DNS_PROVIDER=dns_cf` 时，才回退到内置 Python ACME 引擎
- 如果使用 `dns_ali` 或 `dns_dp`，则明确要求安装 `acme.sh`

### 3. Cloudflare Token 模式是当前内置引擎的唯一目标

内置引擎明确服务于：

- Cloudflare API Token
- DNS-01
- 主域名 + 泛域名证书

这意味着未来如果要支持更多场景，仍需要继续扩展。

## 建议的后续迭代方向

### 继续完善自研最小版 ACME 引擎

当前已经落地了一个自研最小版 ACME 引擎，后续更推荐继续沿着这条路线演进，而不是回到宝塔 `acme_v2.py`。

目标：

- 不依赖宝塔内部 Python 模块
- 不依赖宝塔内部配置文件
- 仅保留本项目需要的最小能力

建议保留的能力范围：

- Cloudflare DNS TXT 验证
- 主域名 + 泛域名证书申请
- 证书续签
- fullchain 与私钥写入本项目的 `CERT_OUTPUT_DIR`
- 与当前 API 状态管理、后台续签、客户端拉取逻辑兼容

这样做的优势：

- 更容易维护
- 更适合开源项目长期演进
- 不受宝塔内部实现变化影响
- 可统一宝塔环境和独立环境的证书行为

### 为什么不建议回到宝塔 `acme_v2.py`

原因：

- 与宝塔内部模块强耦合
- Cloudflare 认证链偏旧
- 后续升级难以同步
- 不利于开源维护

因此更推荐：

- 参考宝塔 ACME 申请流程
- 提炼最小必要能力
- 单独实现本项目自己的兼容引擎

## 当前结论

现阶段：

- `acme.sh` 是 `dns_cf`、`dns_ali`、`dns_dp` 场景下的推荐引擎
- 内置 Python ACME 引擎是无 `acme.sh` 时的正式兜底方案
- 内置 Python ACME 引擎当前仅支持 `dns_cf`
- 宝塔 `acme_v2.py` 不再参与主流程

后续如果要提高项目独立性和可维护性，应优先推进：

- 扩展自研最小版 ACME 引擎
